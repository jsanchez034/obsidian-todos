# Sync OpenSpec config.yaml with agent-skills

Run this prompt whenever the agent-skills plugin or OpenSpec is updated to keep `openspec/config.yaml` aligned with the current skill definitions.

## Goal

Update `openspec/config.yaml` so that artifacts generated by `opsx:propose` encode the actual processes from the installed agent-skills. The config has two sections that reference skills:

- **`context`** — describes skill processes that `opsx:apply` follows when implementing tasks
- **`rules`** — per-artifact rules (proposal, specs, design, tasks) that shape how artifacts reference and enforce skill workflows

## Steps

### 1. Read the current state of both sources

Read these files:

- `openspec/config.yaml` — the file you will update
- All SKILL.md files in the agent-skills plugin:
  ```
  ~/.claude/plugins/marketplace/addy-agent-skills/skills/*/SKILL.md
  ```
- All command files that define the slash-command entry points:
  ```
  ~/.claude/plugins/marketplace/addy-agent-skills/.claude/commands/*.md
  ```
- The OpenSpec customization docs (for valid config.yaml structure):
  - Fetch https://github.com/Fission-AI/OpenSpec/blob/main/docs/customization.md

### 2. Identify what changed

Compare the current `openspec/config.yaml` against the skill definitions. Look for:

- **New skills** added to agent-skills that aren't referenced in the config
- **Removed skills** referenced in the config that no longer exist
- **Changed processes** — skill workflows (steps, cycles, patterns) that have been updated in SKILL.md but are stale in the config
- **Renamed skills** — skill names that changed
- **New artifact types** in OpenSpec schemas that need rules

### 3. Update the `context` block

The context block must describe the core processes from the three primary skills used during `opsx:apply`. For each, extract the actual workflow steps from its SKILL.md:

- **test-driven-development** — the RED-GREEN-REFACTOR cycle and Prove-It pattern (from `skills/test-driven-development/SKILL.md`)
- **incremental-implementation** — the increment cycle and slicing rules (from `skills/incremental-implementation/SKILL.md`)
- **code-review-and-quality** — the review axes and categorization (from `skills/code-review-and-quality/SKILL.md`)

For remaining skills, list them under "Additional skills applied as needed" with a one-line summary extracted from their SKILL.md description field.

**Important**: Use actual skill names (e.g. `test-driven-development`), not slash commands (e.g. `/test`).

### 4. Update the `rules` block

Update rules for each artifact type. The critical constraints:

**`proposal` rules:**

- Must reference skills by their actual names from SKILL.md
- Must require a "Skills needed" section

**`specs` rules:**

- Must connect acceptance criteria to test-driven-development (scenarios should become failing tests)
- Must reference frontend-ui-engineering for accessibility when applicable

**`design` rules:**

- Must require an "Implementation approach" section using skill names and process phases (e.g. "test-driven-development RED")
- Must enforce TDD ordering: RED phases before GREEN phases

**`tasks` rules (most critical):**

- Must enforce test-driven-development ordering: RED tasks (write failing tests) BEFORE GREEN tasks (implement to pass). This is the single most important rule.
- Tasks must describe the process to follow, not just what to build. The task description must include the TDD phase.
- Tasks must be tagged with skill name and phase, e.g. `(test-driven-development: RED)`, `(incremental-implementation)`
- Each task must end with a concrete runnable verification command
- Tasks must follow incremental-implementation: thin vertical slices, each independently verifiable

### 5. Validate

- Confirm the updated file is valid YAML (watch for unescaped backticks, colons in strings, special characters that need quoting)
- Verify every skill name referenced in the config exists in the current SKILL.md files
- Verify no removed skills are still referenced
- Check that the TDD ordering rule is present and clear in the tasks rules

### 6. Report changes

Summarize what was added, removed, or updated so the change is reviewable.

## Design principles

These explain WHY the config is structured this way, so you can make good judgment calls on edge cases:

1. **`opsx:apply` reads task descriptions and implements them directly** — it does not invoke other skills. So the task descriptions themselves must encode the skill processes. This is why the rules are so prescriptive about task content.

2. **Skill names, not slash commands** — the config references `test-driven-development`, not `/test`. Slash commands are entry points that invoke skills, but the config needs to describe the underlying processes.

3. **TDD ordering is non-negotiable** — the most common failure mode is generating implementation tasks before test tasks. The rules must explicitly prevent this.

4. **Process descriptions, not labels** — tagging a task with a skill name is not enough. The task must describe what following that skill means (e.g. "Write failing tests — tests must fail since X does not exist yet" not just "Write tests").
